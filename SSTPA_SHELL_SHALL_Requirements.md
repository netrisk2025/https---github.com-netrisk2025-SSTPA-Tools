| 3.6.3-002 | 3.6.3 Relationship Groups (Nested) | Each Relationship Group SHALL:<br>• Display the relationship name and count<br>• Be collapsible<br>• Include buttons:<br>• Add (create new related node)<br>• Associate (link existing node) |
| 3.6.3-003 | 3.6.3 Relationship Groups (Nested) | For State cards, the TRANSITIONS_TO relationship group SHALL distinguish transitions by TransitionKind so the user can visually identify functional transitions, countermeasure-required transitions, and transitions serving both roles. |
| 3.6.4-001 | 3.6.4 Secondary and Tertiary Entity Cards | Within each Relationship Group, related entities SHALL be displayed as cards.<br>• Secondary entity cards MAY expand to show their own relationship groups (tertiary level)<br>• This pattern SHALL recursively apply to all supported hierarchy levels |
| 3.6.5-001 | 3.6.5 Repeated Entity Representation | • All instances SHALL reference the same underlying node (via HID and uuid)<br>• Editing any instance SHALL update the single underlying node<br>• Each instance SHALL display its HID to preserve identity clarity |

| 3.7.5.1-002 | 3.7.5.1 Commit Notification Behavior | The Frontend SHALL not independently determine final ownership notification recipients. |
| 3.7.5.1-003 | 3.7.5.1 Commit Notification Behavior | The Backend SHALL determine affected owners and generate required messages. |
| 3.7.5.1-004 | 3.7.5.1 Commit Notification Behavior | If owner notification generation fails, the Frontend SHALL display the overall commit failure and no staged changes SHALL be considered committed. |
